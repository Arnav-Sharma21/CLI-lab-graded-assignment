1. DIRECTORY STRUCTURE CREATION
-------------------------------
This section details the creation of the secure project workspace directory tree.

Command Run:
mkdir -p ~/secure_project/docs ~/secure_project/src ~/secure_project/logs

Output:
(No stdout returned; directories successfully created)

Explanation:
The 'mkdir -p' command recursively creates the parent directory 'secure_project' along with its nested child directories 'docs', 'src', and 'logs' in a single command execution.

2. DEFAULT PERMISSIONS (BEFORE STATE)
-------------------------------------
This section details the default directory permissions assigned upon creation before modifications.

Command Run:
ls -ld ~/secure_project ~/secure_project/*

Output:
drwxr-xr-x 5 labex labex 4096 Jun 21 04:10 /home/labex/secure_project
drwxr-xr-x 2 labex labex 4096 Jun 21 04:10 /home/labex/secure_project/docs
drwxr-xr-x 2 labex labex 4096 Jun 21 04:10 /home/labex/secure_project/logs
drwxr-xr-x 2 labex labex 4096 Jun 21 04:10 /home/labex/secure_project/src

Explanation:
The default permission bitmask creates folders with read, write, and search/execute access for the owner, and read and search/execute access for group members and others (755/drwxr-xr-x).

3. RESTRICTED PERMISSIONS (AFTER STATE)
---------------------------------------
This section documents the enforcement of owner-only security permissions on directories and files.

Command Run:
chmod 700 ~/secure_project ~/secure_project/docs ~/secure_project/src ~/secure_project/logs

Output:
(No stdout returned; permissions successfully updated)

Explanation:
The 'chmod 700' command restricts directory permissions to the owner only (rwx------), preventing any other users or groups on the system from accessing or listing their contents.

Command Run:
ls -ld ~/secure_project ~/secure_project/*

Output:
drwx------ 5 labex labex 4096 Jun 21 04:10 /home/labex/secure_project
drwx------ 2 labex labex 4096 Jun 21 04:10 /home/labex/secure_project/docs
drwx------ 2 labex labex 4096 Jun 21 04:10 /home/labex/secure_project/logs
drwx------ 2 labex labex 4096 Jun 21 04:10 /home/labex/secure_project/src

Explanation:
Re-running 'ls -ld' verifies that the directory permissions have changed from 'drwxr-xr-x' (755) to 'drwx------' (700), successfully securing the directories.

4. OWNERSHIP CHANGE AND VERIFICATION
------------------------------------
This section demonstrates how file ownership is explicitly assigned and verified.

Command Run:
chown -R labex:labex ~/secure_project

Output:
(No stdout returned; ownership successfully updated)

Explanation:
The 'chown -R' command recursively sets the user owner and group owner of the workspace to 'labex', ensuring the files are linked to the correct accounts.

Command Run:
ls -laR ~/secure_project | grep -E '^d|^total' -v

Output:
-rw------- 1 labex labex    0 Jun 21 04:10 plain.txt
-rw------- 1 labex labex    0 Jun 21 04:10 app.log
-rwx------ 1 labex labex    0 Jun 21 04:10 app.sh

Explanation:
This filtered long-listing confirms that all files inside the subdirectories are owned by the 'labex' user and group, ensuring only they are granted the designated owner access levels.

5. SYSTEM UMASK CHECK
---------------------
This section details checking the active user shell creation mask.

Command Run:
umask

Output:
0022

Explanation:
The 'umask' command prints the current file mode creation mask, which strips default write access from groups and others (022) for newly created files and directories.

6. SECURITY SUMMARY (4-LINE SUMMARY)
------------------------------------
By setting the root and subdirectories to permission level 700, we completely block unauthorized local users from entering, listing, or navigating the workspace.
Restricting database logs and plain text documents to 600 prevents local processes from reading sensitive operational data or writing malicious payloads.
Setting the source script file to 700 allows the authorized developer to execute code without exposing the execution flow or proprietary logic to other accounts.
These permission controls combine with a restrictive umask configuration to form a robust defense-in-depth framework that keeps project resources safe.
